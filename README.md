### Описание содержимого
##### src/etl_pipeline_script.py

1	Проверяет/создаёт БД dwh_dit_db

2	Читает файлы: customers.csv, orders.json, payments.csv, events.xml, products.xlsx

3	Чистит данные и трансформирует данные: телефоны → +7..., даты → datetime, удаляет дубликаты, меняет типы данных

4	Проверяет внешние ключи: невалидные ID → -1, пустые → 0

5	Сохраняет проблемные строки в bad_records/

6 Сохраняет логи в logs/

7	Загружает данные в схему dds в Greenplum/PostgreSQL

##### ddl/scripts_dwh_star.sql

1 Удаление если есть/Создание схем dim (измерения) и fact (факты)

2	Копирует данные из dds в измерения:
- dim_customer — клиенты
- dim_product — товары
- dim_date — уникальные даты из фактов
  
3	Копирует данные в факты:
- fact_orders — заказы
- fact_events — события
- fact_payments — платежи

### Архитектурные решения

#### 1. Выбор СУБД

Для хранения данных используется Greenplum, так как проект предполагает работу с аналитическими запросами на больших объёмах. Альтернативно, для локальной разработки может использоваться PostgreSQL (совместим с синтаксисом Greenplum). Я же его использовал потому что уже был готовый к работе.

#### 2. Модель данных: звезда (Star Schema)

В качестве DWH-модели выбрана схема «звезда» (Star Schema). Это позволяет:

ускорить аналитические запросы (меньше JOIN-ов),

упростить построение отчётов и витрин,

обеспечить понятную структуру для бизнес-пользователей.

(взял её потому что с этими данными она была уже готова, нужно было только выделить измерения и факты + добавил измерение даты)

#### 3. Слои данных

Данные в базе организованы в три слоя:

dds (Data Delivery Service) — очищенные и подготовленные данные,

dim — измерения (клиенты, товары, даты),

fact — факты (заказы, платежи, события).

mart — витрины представления по запросам задания 4

#### 4. ETL / ELT

В проекте реализован ETL-пайплайн на Python: данные извлекаются из файлов, трансформируются (очистка, приведение типов, проверка FK) и загружаются в слой dds. 

(ETL только потому что в задании было сказано ETL пайплайн.)

#### 5. Обработка ошибок и качество данных

Для контроля качества данных реализованы:
- проверка внешних ключей (замена невалидных ID на -1),
- контроль данных в ключевых столбцах и создание служебных значений (замена данных которых изначально не было на 0, невалидных данных на -1),
- приведение телефонов к единому формату,
- приведение типов данных к единообразным форматам,
- логирование проблемных записей в bad_records/,
- удаление дубликатов.
